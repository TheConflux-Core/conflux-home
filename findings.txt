## Finding 1: Windows-unsafe `block_in_place(block_on)` Pattern — ✅ RESOLVED

**Previous state (pre-fix):**
Both `log_tool_security_event` (tools.rs) and `check_security_gate` (tools.rs) used:
```rust
tokio::task::block_in_place(|| Handle::current().block_on(...))
```
This causes a PANIC on Windows when called from a non-Tokio thread because
`Handle::current()` panics if there is no Tokio runtime in the current thread.

**Current state (verified 2026-04-30):**
Both functions now use the safe pattern:
```rust
std::thread::spawn(move || {
    let rt = tokio::runtime::Runtime::new().unwrap();
    rt.block_on(async_fn(...))
});
```
The spawned thread has its own Tokio runtime. All borrowed references
(agent_id, tool_name, target, db) are captured by move into the closure.

**Verdict:** ✅ No remaining instances of the unsafe pattern in execute functions.
Zero `Handle::current().block_on` calls found in any execute function.

## Finding 3: NO_SCHEMA Tools — Not All Missing Schemas

The baseline CSV marked 14 tools as NO_SCHEMA. Independent audit finds:

**Correctly NO_SCHEMA (no user-facing args, handled internally):**
- `time`, `memory_read`, `memory_write` — handled by runtime
- `dream_get_dashboard`, `echo_counselor_get_state`, `echo_counselor_get_exercises` — internal state tools
- `fridge_shopping_for_meals`, `home_get_appliances` — no input needed
- `kitchen_pantry_heatmap`, `life_morning_brief`, `life_get_heatmap` — dashboard-style

**email_send has args but was marked NO_SCHEMA** — this is actually correct
behavior for execute_email_send: it reads `to`, `subject`, `body` directly from
args.get() regardless of whether the schema mandates them as required. The
schema has them as required; the CSV shows "body|subject|to" in exec_args but
was labeled NO_SCHEMA due to the CSV's rendering of the schema's `properties`
block. Confirmed: email_send's execute function reads all three fields.

**Verdict:** NO_SCHEMA tools are intentional or edge-case rendering artifacts.
No functional gap.

## Finding 4: Hallucinated Tool Name Handling — ✅ Graceful

The dispatch table's catch-all (`_ =>`) does two things:
1. Logs at WARN level: `[tools] UNKNOWN TOOL (possible hallucination): '{name}'`
2. Returns an error ToolResult with the message: `"Unknown tool: 'X'. Use one of the tools listed above."`

This is better than silently returning success or ignoring the call.
No changes recommended.

---

## Finding 5: Direct-Action Behavior — Observation (No Changes Needed)

The tool layer is reactive: the model calls a tool, it executes, a result is
returned. For "direct action" tools like `ui_action`, `notify`, or `web_post`,
this is the right model.

Tools that could benefit from more explicit few-shot examples in the runtime
prompt (so the model knows exactly when and how to call them):
- `ui_action` — the widget/action/value structure is complex; model benefits from examples
- `life_parse_input` — routes natural language to specific tools; model needs examples
- `budget_parse_natural` — same; natural language → structured entry

The runtime.rs already has 7 few-shot examples for general tool calling.
The `ui_action` schema in `get_integration_tool_definitions()` is already rich.

No code changes recommended — this is a prompt engineering observation.
